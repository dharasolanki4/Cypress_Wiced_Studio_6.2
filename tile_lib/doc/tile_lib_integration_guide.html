1. Introduction

This guide will provide a general overview on how to integrate the Tile Library (Tile Lib) to a BLE microcontroller.

This guide assumes the reader has basic BLE knowledge (GAP, GATT, Advertising…).

  1. Advertising Tile Service

Support for Tile Service is advertised through the List of Supported 16-bit UUID in the advertising packet.

The 2 following UUIDs are supported:

  • 0xFEEC: Shipping Mode:
  • This is the mode the Tile is in when in the packaging.
  • In this mode, the Tile is ready to be Activated and associated to a Tile customer account.
  • For applications that don't support activating a Tile, there is usually no need to detect these Tiles.
  • 0xFEED: Activated Mode:
  • This is the mode the Tile will be in most of its active life.
  • A Tile is in its mode after it has been Activated and associated with a Tile customer account.
  • It will then stay in this mode until its death.

  1. Tile Over-the-air API (TOA)

For integrating Tile Lib, it can be useful to understand what TOA is. TOA is the API over BLE that is used to communicate between the Tile app and a Tile Node:

  • TOA is a transport using 2 GATT characteristics.
  • TOA commands are received by the server using GATT Write without Response on Characteristic TOA_CMD.
  • TOA responses are transmitted using GATT Notify on characteristic TOA_RSP.
  • TOA Client sends TOA_CMD and TOA Server sends TOA_RSP.

  1. Getting Accepted by Tile Network

In order to be able to communicate with Tile Network, the following pair needs to be valid:

  • Tile ID.
  • Tile Auth Key.

The values may be hardcoded for integrating and testing. However, they need to be coming from a list of valid registered values.

  1. Tile Lib Minimum Support

The goal of this section is to discuss the minimum amount of code the integrating developer shall implement for the Tile Node to provide minimum interoperability with the Tile app.

In order to achieve minimum TileLib interoperability, a few drivers and modules shall be implemented. For these drivers and modules, a header file describes the requirements and the associated C file needs to be implemented for the target platform. Here is the list of drivers and modules required for TileLib minimal Support:

  1. tile_gap_driver

Provides TileLib Control over GAP functions, like connection, disconnection and connection parameters.

  1. tile_timer_driver

Provides TileLib an interface to use timers.

  1. tile_rand_driver

Provides TileLib an interface to generate random numbers.

  1. tile_tdi_module

Tile Device Information module. Provides TileLib Device specific information and unique numbers.

  1. tile_toa_module

Tile Over-the-air API module. Configures/enables Tile BLE API.

  1. tile_tmd_module

Tile Mode module interface. Controles Tile Mode.

  1. Persistent Storage

Persistent storage is managed by the Application (Not TileLib) and is used to store unique numbers and configuration parameters in a safe way that persists across power failures, reboots and FW Updates.

For minimum TileLib support, storage is not required. However, without storage, the Tile will not be functional after a power failure or reboot.

  1. Audio Player

The audio Player is implemented in the Application (Not in TileLib). The following functions need to be supported:

  • void InitPlayer(void);
  • int PlaySong(uint8_t number, uint8_t strength);
  • int StopSong(void);
  • bool SongPlaying(void);

Then the tile_song_module needs to be implemented so TileLib can play songs.

  1. Button Driver

The tile_button_driver provides TileLib with button press notifications and status.
